## 1. 시스템 개요

### 1.1 서비스 소개

**콘서트 예약 시스템**은 대규모 티켓팅 트래픽을 안정적으로 처리하기 위한 예약 플랫폼입니다.

**핵심 기능:**
- 대기열 기반 트래픽 제어
- 실시간 좌석 조회 및 예약
- 콘서트 랭킹 시스템

### 1.2 기술 스택

| 분류 | 기술 |
|------|------|
| Backend | Java 17, Spring Boot 3.4.1 |
| Database | MySQL 8.0, Redis 7 |
| ORM | JPA/Hibernate |
| 캐시 | Redis (Spring Cache) |
| 분산락 | Redisson |
| 메시징 | Kafka (이벤트 발행) |
| 테스트 | JUnit 5, Testcontainers |

### 1.3 아키텍처

```

│                        Client Layer                             │
└─────────────────────────────┬──────────────────────────────────┘
                              │

                              │

### 1.4 도메인 구조

```
├── concert/          # 콘서트, 스케줄, 좌석 (Layered)
├── reservation/      # 예약 (Clean Architecture)
├── payment/          # 결제 (Clean Architecture)
├── token/            # 대기열 토큰 (Redis)
└── user/             # 사용자 잔액 (Layered)
```

## 2. 주요 API 및 기능

### 2.1 API 목록

| API | Method | Endpoint | 설명 | 캐시 |
|-----|--------|----------|------|------|
| 토큰 발급 | POST | `/api/queue/token` | 대기열 진입 | - |
| 상태 조회 | GET | `/api/queue/status` | 대기열 상태 | - |
| 콘서트 목록 | GET | `/api/concerts` | 전체 목록 | 5분 |
| 콘서트 상세 | GET | `/api/concerts/{id}` | 상세 정보 | 10분 |
| 스케줄 조회 | GET | `/api/concerts/{id}/schedules` | 예약 가능 일정 | 3분 |
| 좌석 조회 | GET | `/api/schedules/{id}/seats` | 예약 가능 좌석 | 10초 |
| 랭킹 조회 | GET | `/api/concerts/ranking` | 인기 순위 | - |

### 2.2 동시성 제어 전략

| 기능 | 전략 | 구현 |
|------|------|------|
| 좌석 예약 | Pessimistic Lock | `SELECT ... FOR UPDATE` |
| 잔액 차감 | Optimistic Lock | `@Version` |
| 토큰 활성화 | Distributed Lock | Redisson |
| 대기열 관리 | Redis Sorted Set | `ZADD`, `ZRANGE` |

```
현재: 단일 인스턴스 (동시 500명)

     │
     ▼ 부하 증가

     ├── App: Load Balancer + 2+ 인스턴스
     ├── DB: Read Replica 추가

```

## 5. 식별된 취약점

### 5.1 병목 지점

| 순위 | 취약점 | 증상 | 영향도 |
|------|--------|------|--------|
| 1 | **토큰 발급 API** | HTTP 500 (고부하 시) | 높음 |
| 2 | DB Connection Pool (3) | 잠재적 고갈 | 중간 |
| 3 | 캐시 Miss 시 DB 부하 | 응답 지연 | 낮음 |

### 5.2 토큰 발급 API 상세

```
문제: 730 동시 사용자에서 3.3% 에러 발생
원인: Redis 동시성 처리 한계 또는 연결 풀 고갈

권장 조치:
1. Rate Limiting 적용 (100 req/s)
2. Redis 연결 풀 증가
3. 배치 처리로 Redis 연산 최적화
```

### 5.3 개선 우선순위

```
┌─────────────────────────────────────────────────────────────┐
│                     개선 로드맵                               │
├─────────────────────────────────────────────────────────────┤
│ [즉시] HikariCP Pool 증가: 3 → 20                           │
│ [즉시] 토큰 발급 Rate Limiting 적용                          │
│ [단기] Redis 연결 풀 최적화                                  │
│ [단기] Slow Query 모니터링 강화                             │

## 7. 결론

### 7.1 강점

- **우수한 읽기 성능**: 캐시 적용으로 평균 17ms 응답
- **안정적인 기본 부하 처리**: 350 동시 사용자, 0% 에러
- **효과적인 대기열**: Redis 기반 실시간 처리

### 7.2 개선 필요

- **토큰 발급 API 안정화**: 고부하 시 에러 발생
- **DB Connection Pool 증가**: 현재 3개로 부족
- **모니터링 강화**: 실시간 알림 체계 구축

### 7.3 권장 사항

| 항목 | 현재 | 권장 |
|------|------|------|
| 동시 사용자 | 500 이하 | 토큰 API 개선 후 확대 |
| HikariCP | 3 | 20 |
| JVM Heap | 512MB | 2GB |
| 인스턴스 | 1 | 대규모 이벤트 시 2+ |

## 8. Q&A

### 자주 묻는 질문

**Q1. 왜 토큰 발급 API에서만 에러가 발생하나요?**
> Redis Sorted Set 연산의 동시성 한계와 연결 풀 경합으로 추정됩니다.
> Rate Limiting과 연결 풀 최적화로 해결 가능합니다.

**Q2. 예상 최대 동시 사용자는?**
> 현재 설정에서는 500명, 스케일 업 후 1,000명, 스케일 아웃 시 2,000명+ 가능합니다.

**Q3. 캐시 전략은 적절한가요?**
> 조회 API는 캐시 효과가 우수합니다. 좌석 조회(10초)는 실시간성과 성능의 균형입니다.
